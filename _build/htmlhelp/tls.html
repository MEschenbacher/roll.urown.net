TLS - Transport Layer Security

We use OpenSSL to generate TLS (Transport Layer Security) certificates and keys (public and private keys).

Following are recommendations valid for April 2014, using Ubuntu 14.04 LTS ‘Trusty Thar’ and OpenSSL 1.0.1f 6 Jan 2014.

Prerequisites

Enough entropy for key generation and encryption. See Provide Entropy.

OpenSSL Configuration

Default system wide OpenSSL configuration is in the file /etc/ssl/openssl.cnf. On a typical server it is mostly used for key generation.

Documentation is available in the online [config(5) man page](https://www.openssl.org/docs/apps/config.html “OpenSSL config”). or as man config on your system.

OpenSSL on a server should behave as follows:

  • Generate RSA private keys with 4096 bits (default is 2048 bits).
  • Generate digital signatures using SHA-2 as digest (default is SHA-1 digest).
  • Don’t encrypt private keys, as servers start and run without any users present to answer password prompts.
  • StartSSL removes all attributes, except the common name from any certificate request.

Generation of keys, certificate signing requests and certificates is best handled on a dedicated desktop system using [xca](http://xca.sourceforge.net/ “XCA - X Certificate and key management”).

CAcert.org

[CAcert.org](http://www.cacert.org) has been dropped from the Ubuntu built-in list of trusted certificate authorities (CA) with the release of Ubuntu 13.10. Also the Debian project no longer includes CAcert root certificates, since it started to use Mozilla’s list of trusted CA’s in 2014.

Steps to include the CAcert root certificate in Ubuntu as trusted CA:

$ sudo mkdir -p /usr/share/ca-certificates/cacert.org
$ cd /usr/share/ca-certificates/cacert.org
$ sudo wget http://www.cacert.org/certs/root.crt
$ sudo dpkg-reconfigure ca-certificates

Key Generation

::
$ export CN=example.com $ openssl genrsa -nodes -out /etc/ssl/private/${CN}.key.pem 4096

Certificate Signing Request (CSR)

::
$ export CN=example.com $ openssl req -out /etc/ssl/${CN}.req.pem -new -key /etc/ssl/private/${CN}.key.pem -sha2

Combined CSR and Key Generation

::
$ export CN=example.com $ openssl req -out /etc/ssl/${CN}.req.pem -newkey rsa:4096 -keyout /etc/ssl/private/${CN}.key.pem -nodes -sha2

Install Signed Certificates

export CN=example.com cat << EOF > /etc/ssl/certs/${CN}.cert.pem —–BEGIN CERTIFICATE—– ... —–END CERTIFICATE—– EOF

Certificate Chains

Certificates signed by [StartSSL](https://startssl.com/) are signed by its intermediary class 1 or class 2 server or client CA.

CAcert certificates may be signed be its intermediary “CAcert Class 3 Root”

TLS clients which verify server certificates expect the server to send the certificates of any intermediary CA along with its own certificate.

On servers this is achieved by providing a certificate-chain-file instead of a certificate file.

Here are the steps to generate such certificate-chain-files.

Download the intermediate CA certificates:

$ cd /etc/ssl/
$ sudo wget -O certs/StartCom_Class_1_Server_CA.pem \
    https://www.startssl.com/certs/class1/sha2/pem/sub.class1.server.sha2.ca.pem
$ sudo wget -O certs/StartCom_Class_2_Server_CA.pem \
    https://www.startssl.com/certs/class2/sha2/pem/sub.class2.server.sha2.ca.pem
$ sudo wget -O certs/CAcert_Class_3_Root.pem \
    http://www.cacert.org/certs/class3.crt

### StartCom Class 1 Primary Intermediate Server CA ###

Assuming the certificate is for the domain “example.com” and our signed certificate is present as /etc/ssl/certs/example.com.crt:

export CN=example.com
cd /etc/ssl/
sudo cat certs/${CN}.crt \
    certs/StartCom_Class_1_Server_CA.pem \
    > ${CN}.chained.crt

### StartCom Class 2 Primary Intermediate Server CA ###

export CN=example.com
cd /etc/ssl/
sudo cat certs/${CN}.crt \
    certs/StartCom_Class_2_Server_CA.pem \
    > ${CN}.chained.crt

### CAcert Class 3 Root ###

export CN=example.com
cd /etc/ssl/
sudo cat certs/${CN}.crt \
    certs/CAcert_Class_3_Root.pem \
    > ${CN}.chained.crt

Diffie-Hellman (DH) Key Exchanges Parameters

To use perfect forward secrecy, Diffie-Hellman parameters must be set up on the server side, otherwise the relevant cipher suites will be silently ignored:

sudo mkdir -p /etc/ssl/dhparams
sudo openssl dhparam -out /etc/ssl/dhparams/dh_1024.pem 1024
sudo openssl dhparam -out /etc/ssl/dhparams/dh_1536.pem 1536

[bettercrypto.org](https://bettercrypto.org) and other sources advise against generating these and instead using proven and properly checked ones and make references to [RFC-3526](https://www.ietf.org/rfc/rfc3526.txt “More Modular Exponential (MODP) Diffie-Hellman groups for Internet Key Exchange (IKE)”).

Unfortunately neither source nor the RFC tells how to get them.

The bettercrypto.org [Git-Repository](https://github.com/BetterCrypto/Applied-Crypto-Hardening) contains a directory with some files and a readme in the [/tools/dhparams](https://github.com/BetterCrypto/Applied-Crypto-Hardening/tree/master/tools/dhparams) directory.

Do:

sudo mkdir -p /etc/ssl/dhparams
sudo wget -O /etc/ssl/dhparams/dh_2048.pem \
    https://git.bettercrypto.org/ach-master.git/blob_plain/HEAD:/tools/dhparams/group14.pem
sudo wget -O /etc/ssl/dhparams/dh_3072.pem \
    https://git.bettercrypto.org/ach-master.git/blob_plain/HEAD:/tools/dhparams/group15.pem
sudo wget -O /etc/ssl/dhparams/dh_4096.pem \
    https://git.bettercrypto.org/ach-master.git/blob_plain/HEAD:/tools/dhparams/group16.pem
sudo wget -O /etc/ssl/dhparams/dh_6144.pem \
    https://git.bettercrypto.org/ach-master.git/blob_plain/HEAD:/tools/dhparams/group17.pem
sudo wget -O /etc/ssl/dhparams/dh_8192.pem \
    https://git.bettercrypto.org/ach-master.git/blob_plain/HEAD:/tools/dhparams/group18.pem
